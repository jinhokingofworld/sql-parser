# Research

## 분석 목적
- `plan.md`에 맞춰 구현했는데도 실제 CLI 실행에서 왜 실패했는지 원인을 정리한다.
- 실패가 "설계 미준수" 때문인지, "계획의 빈틈" 때문인지, "환경 차이" 때문인지 구분한다.

## 결론 요약
- 핵심 실패 원인은 `plan.md`의 큰 방향이 틀려서가 아니라, 실제 실행 환경과 파일 경계 조건을 계획 단계에서 충분히 명시하지 않았기 때문이다.
- 즉, 아키텍처 방향은 맞았지만 "운영 환경 호환성"과 "실제 CSV 파일 상태"에 대한 방어 로직이 초기 구현에 부족했다.

## 실제 발생한 실패와 원인

### 1. `Exec format error`
증상:
- `./sql_processor --sql ...` 실행 시 `cannot execute binary file: Exec format error`

원인:
- 바이너리가 현재 실행 중인 Linux 컨테이너에서 빌드된 것이 아니었다.
- 다른 환경에서 생성된 실행 파일을 그대로 컨테이너에서 실행하려고 해서 생긴 문제였다.

왜 plan 단계에서 놓쳤는가:
- `plan.md`에는 CLI 실행 흐름과 빌드 목표는 있었지만, "실행 파일은 현재 환경에서 다시 빌드해야 한다"는 운영 가정이 명시되지 않았다.
- 즉, 코드 구조 문제가 아니라 배포/실행 환경 문제였다.

판단:
- 구현 자체의 기능 버그라기보다는 실행 환경 차이에서 발생한 문제

### 2. Linux 컨테이너에서 `PATH_MAX` 빌드 실패
증상:
- `src/schema.c`, `src/storage.c`에서 `PATH_MAX undeclared`

원인:
- 구현에서 경로 버퍼 크기를 `PATH_MAX`에 의존했다.
- 하지만 현재 Linux 컨테이너/컴파일 환경에서는 해당 매크로가 보장되지 않았다.

왜 plan 단계에서 놓쳤는가:
- `plan.md`에는 디렉터리 구조와 파일 경로 설계는 있었지만, 경로 버퍼 구현 세부사항의 이식성 요구는 없었다.
- "C로 구현"이라는 조건은 있었지만, "macOS/Linux 컨테이너 모두 빌드 가능해야 한다"는 명시적 체크리스트가 없었다.

판단:
- 구현 세부의 이식성 부족

대응:
- `PATH_MAX` 대신 `SQL_PATH_BUFFER_SIZE` 상수를 공통 헤더에 정의해 고정 버퍼를 사용하도록 수정

### 3. `row column count mismatch: expected 3 fields, got 5`
증상:
- `INSERT` 후 바로 `SELECT`를 수행할 때 CSV 파싱이 깨짐

원인:
- 기존 `data/tables/users.csv`의 마지막 줄 끝에 개행이 없었다.
- 그 상태에서 `append`가 일어나면서 새 row가 다음 줄이 아니라 기존 마지막 줄 뒤에 붙었다.
- 결과적으로 `25,서원규,403,Charlie,19` 같은 잘못된 한 줄이 만들어졌다.

왜 plan 단계에서 놓쳤는가:
- `plan.md`에는 `INSERT -> 파일에 append`, `CSV escape / unescape`는 있었지만,
  아래 조건이 명시되지 않았다.
  - 기존 파일 마지막 줄에 개행이 없을 수 있음
  - append 전에 row boundary를 보장해야 함
- 즉, CSV 포맷을 선택했지만 실제 파일의 불완전한 상태까지는 계획에 포함되지 않았다.

판단:
- 파일 기반 저장 구현에서 row boundary 방어 부족

대응:
- append 전에 파일 끝 문자가 개행인지 확인하고, 아니면 먼저 `\n`을 넣는 `ensure_row_boundary()` 추가

### 4. `row column count mismatch: expected 3 fields, got 1`
증상:
- 위 문제 수정 후 다시 실행했을 때 빈 줄을 1컬럼 row처럼 읽으며 실패

원인:
- CSV 파일 안의 빈 줄을 정상 데이터 row로 해석했다.
- `read_csv_rows()`가 줄바꿈 제거 후 길이 0인 줄을 건너뛰지 않았다.

왜 plan 단계에서 놓쳤는가:
- `plan.md`에는 "엣지 케이스 고려"는 있었지만, 저장소 레벨에서의 구체적인 비정상 데이터 조건이 세분화되어 있지 않았다.
- 특히 아래 항목이 빠져 있었다.
  - 빈 줄 무시
  - 손상된 CSV 라인 처리
  - trailing newline/blank line 허용 여부

판단:
- storage 레이어의 입력 정제 부족

대응:
- 빈 줄은 건너뛰도록 `read_csv_rows()` 수정

## plan 대비 부족했던 부분

### 1. 환경 호환성 체크 부족
- `plan.md`는 구조적으로는 타당했지만, 빌드 환경 차이에 대한 검증 계획이 없었다.
- 최소한 아래 항목이 있었다면 초기에 잡을 수 있었다.
  - macOS와 Linux 컨테이너에서 각각 빌드 확인
  - 비표준 매크로/플랫폼 의존 API 사용 여부 점검

### 2. storage 레이어의 실제 파일 상태 가정이 너무 낙관적이었음
- 계획은 CSV를 "정상적인 줄 단위 파일"로 가정했다.
- 하지만 실제 파일 기반 DB에서는 다음 상황이 흔하다.
  - 마지막 줄 끝 개행 누락
  - 중간 blank line 존재
  - 일부 라인 손상
- 이 부분이 초기 구현과 테스트 설계에 충분히 반영되지 않았다.

### 3. integration 테스트가 fixture 기준으로만 검증되었음
- 테스트용 fixture는 깨끗한 CSV를 사용해서 모두 통과했다.
- 실제 `./data` 파일처럼 개행이 없거나 빈 줄이 있는 상태는 테스트하지 않았다.
- 그래서 "테스트는 통과하지만 실제 데이터에서는 실패"가 발생했다.

## 재발 방지 제안

### 테스트 보강
- storage 단위 테스트에 아래 케이스 추가
  - 마지막 줄 개행 없는 CSV에 append
  - 빈 줄이 포함된 CSV 읽기
  - 손상된 라인 감지
- integration 테스트에 fixture 외 실제와 유사한 "dirty CSV" 케이스 추가

### plan 보강 문장 제안
- `storage` 항목에 다음을 추가하면 좋다.
  - "append 전 기존 파일의 마지막 row 경계를 보장한다."
  - "읽기 시 blank line을 무시하고, 손상된 row는 명확한 에러로 보고한다."
- `품질 테스트` 항목에 다음을 추가하면 좋다.
  - "운영 데이터의 비정상 포맷(마지막 줄 개행 누락, 빈 줄, 손상 row)을 테스트한다."
  - "Linux 컨테이너 환경에서 빌드 및 실행을 검증한다."

## 최종 판단
- 프로그램이 처음 실패한 이유는 `plan.md` 자체가 틀렸기 때문은 아니다.
- 실패의 본질은 아래 3가지다.
  - 실행 환경 차이
  - 이식성 고려 부족
  - file-based storage의 경계 조건 테스트 부족
- 따라서 이번 사례는 "설계 방향 오류"보다는 "운영 디테일과 품질 검증 범위 부족"에 가까운 실패였다.
