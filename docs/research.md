# Research

## 관련 파일
- agent.md

## 현재 저장소 상태
1. 저장소에는 현재 `agent.md`만 존재한다.
2. `src/`, `include/`, `tests/`, `Makefile` 등 구현 파일은 아직 없다.
3. 따라서 기존 코드 흐름을 분석하는 대신, 과제 요구사항을 기준으로 초기 아키텍처를 설계해야 한다.

## 현재 동작 흐름
1. 아직 CLI 입력 경로가 구현되어 있지 않다.
2. 아직 SQL 파서가 구현되어 있지 않다.
3. 아직 실행기(executor)와 파일 저장소(storage)가 구현되어 있지 않다.

## 요구사항 해석
- 입력 방식: 텍스트 파일에 작성된 SQL을 CLI 명령으로 전달해야 한다.
- 최소 지원 SQL:
  - `INSERT`
  - `SELECT`
- 실행 요구:
  - `INSERT`는 파일에 저장
  - `SELECT`는 파일에서 읽어서 출력
- 저장소 제약:
  - 테이블 단위 파일 관리
  - 스키마와 테이블은 이미 존재한다고 가정
- 품질 요구:
  - 단위 테스트
  - 기능 테스트
  - 엣지 케이스 고려
  - 포트폴리오 수준 완성도

## 핵심 설계 대상
- CLI 입력 처리 위치
- SQL tokenizer / parser 설계
- AST 또는 명령 구조체 설계
- executor 흐름
- file-based storage 포맷
- schema metadata 관리 방식
- 테스트 전략

## 발견 사항
- 기존 코드가 없으므로, 설계 자유도가 높다.
- 반대로 초기 구조를 잘못 잡으면 이후 확장이 어려워질 수 있다.
- `SELECT`와 `INSERT`만 요구되므로, 전체 SQL 엔진보다 "작고 선명한 처리기"로 설계하는 것이 적절하다.

## 문제점 / 리스크
- 스키마가 이미 존재한다고 가정하지만, 스키마 정보를 어디서 읽을지 명시되어 있지 않다.
- CSV를 선택하면 구현은 쉽지만, 문자열 이스케이프와 쉼표 처리 난도가 올라간다.
- SQL 파싱을 단순 split으로 처리하면 공백, 대소문자, 문자열 리터럴에서 쉽게 깨질 수 있다.
- 초기 구현에서 WHERE 절을 완전히 제외할지, 제한적으로 포함할지 명확히 정해야 한다.

## 권장 방향
- 1차 목표는 `INSERT INTO ... VALUES ...` 와 `SELECT ... FROM ... [WHERE column = value]`를 안정적으로 처리하는 것이다.
- 파싱은 "토큰 기반"으로 시작하되, 문법 범위는 제한해 복잡도를 관리한다.
- 저장 포맷은 CSV보다 "행 단위 custom text + schema metadata" 또는 TSV 계열이 안정적이다.
- 테스트 가능한 구조를 위해 `parser`, `executor`, `storage`, `schema`, `cli`를 분리한다.

## 요약 판정
- 전체적으로 `plan.md`는 과제의 핵심 구현 범위를 잘 반영하고 있다.
- 특히 `CLI 입력 -> 파싱 -> 실행 -> 파일 저장` 흐름, `INSERT`/`SELECT`, 파일 기반 저장, 테스트 전략은 명확하다.
- 다만 엄격하게 보면 아래 항목은 `부분 반영`으로 보는 것이 맞다.
  - `memo.md`의 "AI를 활용해도 되지만 핵심 로직은 이해하고 설명할 수 있어야 한다"는 운영 원칙이 `plan.md`에 명시적 완료 기준으로 들어가 있지 않다.
  - 디렉터리 구조는 기본 확장성은 있으나, 기능이 늘어나면 `parser.c`, `executor.c`, `storage.c`가 비대해질 가능성이 있다.
  - 품질 테스트 계획은 포함되어 있으나, C 프로젝트 특성상 메모리 안정성 검증 계획까지 있으면 더 완전하다.

## 엄격 평가 시 특이사항

### 1. 부분 반영 항목
- `memo.md:78-84`
  - 현재 계획은 "이해하기 쉬운 구조"와 "설명 가능한 AST 출력"은 포함한다.
  - 하지만 "AI를 써도 되지만 생성된 코드의 핵심 로직을 팀원이 설명 가능해야 한다"는 항목이 별도 체크리스트나 완료 기준으로 적혀 있지는 않다.
  - 따라서 평가상 `부분 반영`으로 기록한다.

### 2. 테스트 계획은 존재하지만, C 품질 기준으로는 보강 여지가 있다
- `plan.md:504-531`에 단위/기능/엣지 테스트 계획은 분명히 존재한다.
- 다만 C 프로젝트 관점에서 아래 항목이 추가되면 더 강해진다.
  - 메모리 누수 검사(`valgrind` 또는 sanitizer)
  - 잘못된 schema 파일 형식 테스트
  - 빈 테이블 파일/손상된 CSV 라인 복구 또는 실패 처리 테스트
  - 여러 SQL 문장을 한 파일에 넣었을 때의 회귀 테스트

## 디렉터리 구조 확장성 평가

### 현재 제안의 장점
- `include/`, `src/`, `tests/`, `data/`, `docs/` 분리가 명확하다.
- `schema`, `storage`, `parser`, `executor`, `cli`로 책임을 나눈 점은 과제 요구와 잘 맞는다.
- `tests/unit`, `tests/integration`, `tests/fixtures` 분리는 기본적인 테스트 확장성을 갖춘 구조다.
- `docs/architecture.md`를 두는 방향은 포트폴리오와 팀 설명 측면에서 좋다.

### 확장성 리스크
- `src/parser.c`, `src/executor.c`, `src/storage.c`가 단일 파일로 계속 커질 가능성이 높다.
  - `UPDATE`, `DELETE`, `CREATE TABLE`, 다중 statement, 표현식 지원이 들어오면 병목이 될 수 있다.
- `src/utils.c`가 있는데 대응되는 `include/utils.h`가 구조 예시에 없다.
  - 구현이 커질수록 인터페이스 경계가 흐려질 수 있다.
- integration 테스트 구조에 SQL 입력 파일만 있고 expected output 또는 test runner 스크립트가 없다.
  - 테스트 수가 늘어나면 검증 자동화가 불편해질 수 있다.
- 런타임 데이터(`data/`)와 테스트용 데이터가 섞일 가능성이 있다.
  - 현재 예시상 `data/tables/users.csv`는 실사용 데이터이고 `tests/fixtures/sample_users.schema`는 테스트 데이터다.
  - 추후 테스트가 실제 `data/`를 건드리면 재현성과 독립성이 떨어질 수 있다.

### 확장성 판정
- 결론: "기본 확장 가능성은 포함되어 있다."
- 다만 장기 확장까지 고려하면 아래 보완이 권장된다.
  - `src/parser/insert_parser.c`, `src/parser/select_parser.c` 같은 기능별 분리
  - `src/executor/insert_executor.c`, `src/executor/select_executor.c` 분리
  - `tests/expected/` 또는 `tests/golden/` 디렉터리 추가
  - 테스트 전용 DB 루트(`tests/tmpdb/` 또는 `tests/fixtures/db/`) 분리
  - `include/utils.h` 명시
