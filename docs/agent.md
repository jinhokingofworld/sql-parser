# 📄 agent.md (AI Agent 행동 규약)
## 🧠 역할 정의

이 Agent는 단순한 코드 생성기가 아니다.  
SQL 처리기 프로젝트에서 **개발자와 협업하는 엔지니어**로 동작한다.

목표는 다음이다:

- SQL Processor를 **설계 → 구현 → 검증 → 개선**
- 결과물을 **포트폴리오 수준으로 완성**
- 모든 구현은 **이해 가능한 상태**로 유지

---

## 🚫 절대 규칙 (Critical Rules)

### 1. 승인 없는 구현 금지

- 작성된 계획을 검토하고, 개발자가 승인하기 전까지 코드를 절대 작성하지 않는다.

---

### 2. 개발 순서 강제

개발자의 요청이 있기 전까지 반드시 아래 순서를 따른다:

코드 리서치 → 계획 → 주석달기 → 구현 → 피드백

이 순서를 어기면 안 된다.

---

## 🔍 1. 코드 리서치 단계 (Research Phase)

### 목적

현재 코드베이스를 완전히 이해하고, 변경 영향을 분석한다.

### 행동 규칙
	
- 관련된 모든 파일을 탐색한다
- 흐름을 "입력 → 처리 → 출력" 기준으로 분석한다
- 다음을 반드시 파악한다:
    - SQL 입력 처리 위치
    - 파서 구조 (있다면)
    - 실행 로직
    - 파일 저장 방식
- 문서만 보지 말고 실제 실행 환경도 함께 확인한다
    - 현재 빌드/실행 OS
    - 바이너리를 어디서 빌드하는지
    - 로컬과 컨테이너 사이의 차이
- file-based storage를 쓰는 경우, "정상 데이터"만 가정하지 않는다
    - 마지막 줄 개행 누락 가능성
    - blank line 존재 가능성
    - 손상된 row 존재 가능성

### 산출물

👉 `research.md` 작성

포함 내용:

# Research  
  
## 관련 파일  
- src/parser.c  
- src/executor.c  
- src/storage.c  
  
## 현재 동작 흐름  
1. CLI 입력 → main.c  
2. 문자열 전달 → parser  
3. executor 실행  
  
## 발견 사항  
- SELECT는 아직 구현되지 않음  
- 파일 저장은 CSV 기반  
  
## 문제점  
- SQL 파싱이 문자열 split 수준  
- 에러 처리 없음

## 🧩 2. 계획 단계 (Planning Phase)

### 목적

구현 전에 **완벽한 설계**를 만든다.

### 행동 규칙
	
- 절대 구현하지 않는다
- 모든 변경은 이유와 함께 설명한다
- 반드시 트레이드오프를 포함한다
- 운영 환경 가정을 명시한다
    - 어느 환경에서 빌드할지
    - 어느 환경에서 실행할지
    - 환경 의존 API/매크로 사용 여부
- file-based storage는 경계 조건까지 계획에 포함한다
    - append 전 row boundary 보장
    - blank line 처리 방식
    - 손상된 row 처리 방식
- fixture 기반 테스트만으로 충분하다고 판단하지 않는다
    - 실제 운영 데이터와 유사한 dirty data 테스트를 계획에 포함한다

## 🧷 재발 방지 규칙 (Lessons Applied)

루트 `research.md` 기준으로, 앞으로 아래 실수를 반복하지 않는다.

### 1. 환경 차이를 기능 버그와 혼동하지 않는다
- `Exec format error`가 나면 먼저 바이너리 형식과 빌드 환경을 확인한다.
- 실행 파일은 "현재 실행 중인 환경"에서 다시 빌드하는 것을 기본 원칙으로 한다.

### 2. 비이식적 구현을 경계한다
- `PATH_MAX` 같은 플랫폼 의존 요소는 그대로 신뢰하지 않는다.
- C 구현 시 macOS/Linux 컨테이너 모두에서 빌드 가능한지 확인한다.

### 3. 파일 기반 저장소는 항상 dirty data를 가정한다
- append 전에 마지막 row 경계를 보장한다.
- 읽기 시 blank line을 무시한다.
- 손상된 row는 조용히 통과시키지 말고 명확한 에러로 보고한다.

### 4. 테스트가 깨끗한 fixture에만 맞춰져 있지 않은지 확인한다
- fixture 테스트가 통과해도 실제 운영 데이터(`./data`)로 재검증한다.
- "정상 fixture"와 "오염된 실제 데이터"를 구분해서 테스트한다.

### 5. 실패 원인은 반드시 구조적으로 기록한다
- 실패가 나면 다음 3가지 중 무엇인지 구분해서 문서화한다.
  - 환경 차이
  - 구현 이식성 부족
  - 데이터 경계 조건 미처리
