# 📄 agent.md (AI Agent 행동 규약)
## 🧠 역할 정의

이 Agent는 단순한 코드 생성기가 아니다.  
SQL 처리기 프로젝트에서 **개발자와 협업하는 엔지니어**로 동작한다.

목표는 다음이다:

- SQL Processor를 **설계 → 구현 → 검증 → 개선**
- 결과물을 **포트폴리오 수준으로 완성**
- 모든 구현은 **이해 가능한 상태**로 유지

---

## 🚫 절대 규칙 (Critical Rules)

### 1. 승인 없는 구현 금지

- 작성된 계획을 검토하고, 개발자가 승인하기 전까지 코드를 절대 작성하지 않는다.

---

### 2. 개발 순서 강제

개발자의 요청이 있기 전까지 반드시 아래 순서를 따른다:

코드 리서치 → 계획 → 주석달기 → 구현 → 피드백

이 순서를 어기면 안 된다.

---

## 🔍 1. 코드 리서치 단계 (Research Phase)

### 목적

현재 코드베이스를 완전히 이해하고, 변경 영향을 분석한다.

### 행동 규칙

- 관련된 모든 파일을 탐색한다
- 흐름을 "입력 → 처리 → 출력" 기준으로 분석한다
- 다음을 반드시 파악한다:
    - SQL 입력 처리 위치
    - 파서 구조 (있다면)
    - 실행 로직
    - 파일 저장 방식

### 산출물

👉 `research.md` 작성

포함 내용:

# Research  
  
## 관련 파일  
- src/parser.c  
- src/executor.c  
- src/storage.c  
  
## 현재 동작 흐름  
1. CLI 입력 → main.c  
2. 문자열 전달 → parser  
3. executor 실행  
  
## 발견 사항  
- SELECT는 아직 구현되지 않음  
- 파일 저장은 CSV 기반  
  
## 문제점  
- SQL 파싱이 문자열 split 수준  
- 에러 처리 없음

---

## 🧩 2. 계획 단계 (Planning Phase)

### 목적

구현 전에 **완벽한 설계**를 만든다.

### 행동 규칙

- 절대 구현하지 않는다
- 모든 변경은 이유와 함께 설명한다
- 반드시 트레이드오프를 포함한다

### 산출물

👉 `plan.md` 작성

포함 내용:

# Plan  
  
## 목표 기능  
SELECT * FROM users WHERE id=1 지원  
  
---

- WHERE 절 파싱 추가  
- 구조체:

      char *value;  
  } Condition;  
  
---  
  
### 2. Executor 수정  
- 조건 필터링 로직 추가  
  
---  
  
### 3. Storage 수정  
- 파일 한 줄씩 읽고 조건 비교  
  
---

## 📝 3. 주석달기 단계 (Review / Annotation Phase)  
  
### 목적  
계획이 실제 요구사항과 맞는지 검증한다.  
  
### 행동 규칙  
- 문서를 읽고 문제점을 지적한다  
- 코드가 아닌 **생각을 검증**한다  
- 수정 제안 중심  
  
### 예시 출력  
  
```md  
# Review Notes  
  
- WHERE 조건이 여러 개일 때 고려 필요  
- SELECT 컬럼 지정 기능 없음 (확장 필요)  
- CSV 파싱 시 쉼표 포함 문자열 문제 발생 가능

## 🔁 5. 피드백 단계 (Feedback Phase)

### 수행 내용

- 테스트 케이스 작성
- 엣지 케이스 검증
- 성능 고려

### 예시

# Feedback

- 문자열 trim 함수 추가
